# Day 6 — Synthetic settlements dataset

**16 NDJSON files · ~48,000 rows · ~15 MB on disk**

Generated by `scripts/generate_m6_settlements.py`. Reproducible (fixed seed = 20260518).

## Layout (Hive-style)

```
data/m6-settlements/
    source_system=BSP/
        ingest_date=2026-05-09/part-00000.jsonl    (3,000 rows · ~950 KB)
        ingest_date=2026-05-10/part-00000.jsonl
        ingest_date=2026-05-11/part-00000.jsonl
        ingest_date=2026-05-12/part-00000.jsonl
    source_system=ARC/
        ingest_date=2026-05-09/part-00000.jsonl
        ingest_date=2026-05-10/part-00000.jsonl
        ingest_date=2026-05-11/part-00000.jsonl
        ingest_date=2026-05-12/part-00000.jsonl
    source_system=ATPCO/
        (same 4 dates)
    source_system=GDS/
        (same 4 dates)
```

4 source systems × 4 ingest dates = 16 partitions × 3,000 rows = **48,000 rows total**.

## Schema (matches Day 6 v3 catalog)

Each NDJSON line is one settlement:

```json
{
  "settlement_id": "BSP-2026-W19-AI-100042",
  "airline_code":  "AI",
  "pnr":           "VWXYZ2",
  "fare_class":    "Y",
  "od_pair":       "BOM-LHR",
  "ticket_number": "098-0000000100042",
  "bsp_cycle":     "2026-W19",
  "issue_date":    "2026-05-09T11:42:00+05:30",
  "fare_amount":   2845.0,
  "currency":      "INR",
  "commission_pct": 0.07,
  "refund_status": null,
  "agency_iata":   "14305820"
}
```

13 columns + 2 partition keys (`source_system`, `ingest_date`) discovered by Crawler.

## How to use

**Option 1 — pull from repo + upload to S3:**

```bash
git pull
bash scripts/upload_m6_settlements.sh <learner>
```

That uploads everything under `validated/<learner>/validated_settlement_json/` matching the Hive layout above. Day 6 Lab 1 step 3 just runs this script and skips the manual re-staging.

**Option 2 — regenerate (if you want fresh data):**

```bash
python3 scripts/generate_m6_settlements.py
```

Reproducible — re-running produces byte-identical output unless the seed changes.

## Realism notes

- **Currencies** match airline operating-currency convention (AI→INR, BA→GBP, EK→AED, etc.).
- **Ticket prefixes** match real IATA source assignments (098=BSP, 104=ARC, 081=ATPCO, 176=GDS).
- **Fare distributions** are class-weighted: Y/M/J/F realistically priced at $850/$1200/$4800/$9500 base ±spread.
- **Commission rates** vary by source: BSP ~7%, ARC ~9%, ATPCO ~5%, GDS ~6.5% (lab approximations).
- **Refund_status** is mostly NULL (real-world refunds are minority); ~30% have REQUESTED/PROCESSED/REJECTED.

This dataset is **synthetic** — not real BSP / ARC / ATPCO / GDS production data. No PII.

## Why 48K rows, not 50K?

Just clean partition arithmetic: 4 sources × 4 dates × 3,000 rows = 48,000. Slide 17 says "≤50K rows fits in Lambda" — 3K per partition is well under that ceiling. The Day 6 ETL Lambda processes one partition at a time, so per-invocation it sees 3K rows, not 48K.

## What this means for the A/B query (Lab 2)

Each partition file = ~950 KB JSON → ~50 KB Parquet (Snappy) after ETL. A query with `WHERE ingest_date='2026-05-09' AND source_system='BSP'` scans:

| Format | Actual on-disk | DataScannedInBytes (Athena bills 10 MB minimum) |
|---|---|---|
| JSON | ~950 KB | **10 MB** (billed minimum) |
| Parquet | ~50 KB on disk; ~5 KB for `fare_amount` column chunk | **10 MB** (billed minimum) |

At lab scale, both hit Athena's 10 MB floor. **The 100× ratio shows up in *production-scale* data (hundreds of partitions, GB per partition)** — at lab scale, the meaningful ratio is the cross-partition scan:

```sql
-- This scans ALL 16 partitions if no WHERE filter
SELECT SUM(fare_amount) FROM accelya_validated_<learner>.validated_settlement_json;
```

JSON: 16 × 950 KB ≈ 15 MB billed.
Parquet: 16 × 5 KB ≈ 10 MB billed (still the floor).
Ratio: ~1.5× at this scale.

**For an honest 100× demonstration**, query without partition filter on JSON; the Athena workgroup's 100 MB cap won't fire on this small a dataset, so the metric reads cleanly. Document both observed and projected ratios in `/reviews/m6-query-cost-evidence.md`.
